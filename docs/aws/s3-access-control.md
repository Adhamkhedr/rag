# S3 Access Control

## Overview

Amazon S3 provides multiple mechanisms to control access to buckets and objects. Properly configuring S3 access control is critical for security — misconfigured S3 buckets are one of the most common causes of data exposure incidents in AWS environments.

## Bucket Policies

Bucket policies are resource-based JSON policies attached directly to an S3 bucket. They define what actions are allowed or denied for which principals on the bucket and its objects. Bucket policies can grant access to IAM users, roles, other AWS accounts, or even anonymous (public) access.

Example: Allow read-only access from a specific account:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCrossAccountRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::111122223333:root"
      },
      "Action": ["s3:GetObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

Bucket policies have a size limit of 20 KB. For complex access requirements, consider using S3 Access Points.

## Access Control Lists (ACLs)

ACLs are a legacy access control mechanism that predates IAM policies. Each bucket and object has an associated ACL that defines which AWS accounts or predefined S3 groups are granted access and the type of access. ACL grantees include: the bucket owner, other AWS accounts (by canonical user ID), the `AuthenticatedUsers` group (any authenticated AWS user), the `AllUsers` group (anonymous/public access), and the `LogDelivery` group.

AWS recommends disabling ACLs for most use cases. As of April 2023, new buckets have ACLs disabled by default (using the `BucketOwnerEnforced` setting for Object Ownership). With ACLs disabled, access is controlled entirely through policies. This simplifies access management and prevents accidental public access through ACL grants.

## Block Public Access

S3 Block Public Access provides account-level and bucket-level settings that override individual policies and ACLs to prevent public access. The four settings are:

- **BlockPublicAcls**: Rejects PUT requests that include a public ACL and rejects PUT Object requests if the bucket has a public ACL.
- **IgnorePublicAcls**: Causes S3 to ignore all public ACLs on the bucket and its objects.
- **BlockPublicPolicy**: Rejects PUT bucket policy requests if the policy grants public access.
- **RestrictPublicBuckets**: Restricts access to a bucket with a public policy to only AWS service principals and authorized users within the bucket owner's account.

AWS recommends enabling all four Block Public Access settings at the account level unless you have a specific need for public access (such as hosting a static website). Account-level settings override bucket-level settings.

## S3 Access Points

S3 Access Points are named network endpoints attached to a bucket that simplify data access for shared datasets. Each access point has its own access point policy, DNS name, and network origin controls. Key features:

- Each access point enforces a distinct set of permissions through its access point policy. This is simpler than managing a single complex bucket policy.
- Access points can be restricted to VPC-only access, preventing internet-based access.
- An access point ARN format: `arn:aws:s3:us-east-1:123456789012:accesspoint/my-access-point`.
- You can create up to 10,000 access points per Region per account.

Access points are ideal for multi-tenant applications where different teams or services need different levels of access to objects in the same bucket.

## Presigned URLs

Presigned URLs grant temporary, time-limited access to a specific S3 object without requiring the requester to have AWS credentials. The URL is generated by a user with existing permissions and embeds authentication information as query string parameters.

Presigned URLs can be created for both `GET` (download) and `PUT` (upload) operations. The maximum expiration is 7 days when generated using IAM user credentials. If the generating identity loses its permissions before the URL expires, the URL stops working.

Example use cases: sharing a private file with an external party, allowing a web application user to upload a file directly to S3, generating download links in emails.

Presigned URLs inherit the permissions of the identity that created them. Never generate presigned URLs using overly broad credentials.

## Cross-Account Access

There are three common approaches for cross-account S3 access:

1. **Bucket Policy**: The bucket owner adds a bucket policy granting access to the other account. The other account's IAM users/roles also need identity-based policies allowing the S3 actions.
2. **IAM Role**: Create a role in the bucket owner's account with S3 permissions and a trust policy allowing the other account to assume it. The external account's users assume the role to access the bucket.
3. **S3 Access Points**: Create an access point with a policy that grants access to the cross-account principal.

For cross-account object uploads, the uploader's account owns the object by default (unless the bucket uses `BucketOwnerEnforced` object ownership). With `BucketOwnerEnforced`, the bucket owner automatically owns all objects regardless of who uploaded them.

## Evaluating S3 Access

When a request is made to S3, the following are evaluated:
1. S3 Block Public Access settings
2. Bucket policies
3. ACLs (if enabled)
4. IAM identity-based policies on the requester
5. VPC endpoint policies (if the request comes through a VPC endpoint)

Access is granted only if all applicable policies allow the request and no applicable policy explicitly denies it.
